# 設計仕様：02-gstack-overview.md 教材リストラクチャリング

**日付**: 2026-06-20  
**対象ファイル**: `docs/03-frameworks/02-gstack-overview.md`  
**目的**: 初心者・中級者向けの価値訴求を強化し、読者が「自分のための記事だ」と感じながら読み進められる構成に再設計する

---

## 背景と課題

### 読者プロファイル

- Claude Code や GitHub Copilot を使い始めているが「思い通りに動かない」「雑に動く」と感じている初心者〜中級者
- AIエージェントに全役割を任せることの限界を体感しているが、解決策を知らない状態でこのページに到達する

### 現状の問題

現行ドキュメントは「概要 → 背景（問題提起）→ 実績 → スプリント → スキル一覧 → インストール → 比較」という論理的な順序で書かれている。「背景：単一AIアシスタント問題」セクションで問題提起はされているが、**抽象的な課題説明**にとどまっており、読者の個人的な体験に寄り添っていない。また Garry Tan の実績数値が「gstackとは何か」より先に来るため、「自分には関係ある？」という印象を与えやすい。

### 改善の方針

**アプローチ B：物語構造への全面再構成**を採用する。

- 情報の削除はゼロ（順番の組み替えと冒頭の新規追加のみ）
- 「共感 → 解決策 → Before/After → 詳細 → インストール」という読者の心理フローに沿った順序に変更
- 大胆な再構成を許可（既存セクションの制約なし）
- Mermaid 図3つ → 起動方式の図1つのみ残し、残りは削除（情報は比較表で代替）

## 新しいセクション構成

### ① 共感フック（新規追加）

**目的**: 読者の「AIに全役割を任せる苦しさ」を具体的なシーンで描写し、「これは自分の話だ」と感じさせる。

**内容**:
- 「AIが自分のコードを自分でレビューした」シーン
- 「なぜこうなったのか誰にもわからない」シーン（設計者と実装者が同一）
- 「QAもセキュリティも自己申告だった」シーン（本番でSQLインジェクション発見）
- 締め：「1人の人間に全役割を兼任させているのが問題。人間のチームでは当然やる役割分担がエージェントには存在しない」

**執筆スタイル**: 二人称的な場面描写。セリフと状況を交えた具体的なシーン3つ。

**変更点**: 現行「背景：単一AIアシスタント問題」の内容（問題の本質）を①に吸収し、このセクションは削除する。

**内容**:

**変更点**:

### ③ 同じ依頼で何が変わるのか Before/After（新規追加）

**目的**: gstack 導入前後で同じ依頼がどう変わるかを具体的なターミナル出力で示す。

**内容**:
- Before（gstack なし）：「カレンダーブリーフィングアプリを作りたい」→ 確認なしに即実装・完了
- After（gstack あり）：`/office-hours` → CEO役が6つの質問・要件抽出・設計ドキュメント生成、`/plan-eng-review` → EM役が技術検証、`/qa` → QAリード役がブラウザテスト、`/ship` → リリースエンジニア役がマージ
- 締め：「役割が分かれると、それぞれの専門家が本来の仕事をする」
