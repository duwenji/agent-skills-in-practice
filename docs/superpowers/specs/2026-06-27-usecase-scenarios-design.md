# 教材品質向上：ユースケースシナリオ追加 設計仕様

**日付**: 2026-06-27
**対象**: `agent-skills-in-practice/docs/` 配下の教材ファイル
**バージョン**: 1.0.0

---

## 目的

「スキルを既に使いこなしており、より深く作れるようになりたい上級者」が、各章を読んだとき **「これは自分のプロジェクトで使える」と冒頭で気づける** 学習体験を実現する。

現状の章構成は、スキルの機能説明（What）・操作手順（How）が中心で、**読者が自分の状況と接続する前に手順が始まる**。上級者は既存ワークフローを持っているため、自分のユースケースに当てはまると気づいた瞬間に読むモードが変わる。この認識のきっかけを各章の冒頭に置く。

---

## 変更内容

### 追加するセクション

各章の「## このスキルについて」（または冒頭の概要セクション）の**直後**に以下を挿入する。

```markdown
## こんな状況に刺さる

> 以下のどれかに当てはまったら、このスキルがあなたの問題を解決します。

- **[職種・役割]として**、[具体的な状況と困りごと]なとき
- **[職種・役割]として**、[具体的な状況と困りごと]なとき
- **[職種・役割]として**、[具体的な状況と困りごと]なとき
```

### シナリオの書き方ルール

| ルール              | 内容                                                                                           | 理由                                   |
| ------------------- | ---------------------------------------------------------------------------------------------- | -------------------------------------- |
| **3点セット** | 職種 × 状況 × 困りごとをセットで書く                                                         | 状況だけでは自分ごとにならない         |
| **個数**      | 2〜3個。4個以上は書かない                                                                      | 散漫になり「どれも自分じゃない」となる |
| **起点**      | 「〜な人に向いています」ではなく「〜なときに使う」と書く                                       | 行動起点の表現が実践との接続を強める   |
| **具体性**    | 「チームで開発しているとき」ではなく「PRレビューで重大度の基準を口頭で説明できないとき」と書く | 上級者は抽象表現に反応しない           |

### 適用例（grill-me.md）

```markdown
## こんな状況に刺さる

> 以下のどれかに当てはまったら、このスキルがあなたの問題を解決します。

- **PRレビュアーとして**、「なんとなく気になる」という感覚を重大度付きの言葉にできず、
  指摘の根拠をうまく説明できないとき
- **チームリードとして**、コーディング規約はあるが「どの違反がブロッカーか」の基準が
  メンバーごとにバラバラなとき
- **ジュニア育成担当として**、フィードバックが厳しくなりすぎて心理的安全性を損ないたくないとき
```

## 適用対象と優先順位

### Tier 1（最優先）: 実践スキル章

効果が最大な章。スキルを実際に動かす内容であり、シナリオが書きやすく上級者の行動変容につながりやすい。

| ファイル                                              | シナリオの軸                                                     |
| ----------------------------------------------------- | ---------------------------------------------------------------- |
| `04-skills-in-practice/01-grill-me.md`              | PRレビュアー / チームリード / ジュニア育成担当                   |
| `04-skills-in-practice/02-triage-issue-analysis.md` | Issue管理者 / OSS メンテナ / チームリード                        |
| `04-skills-in-practice/03-improve.md`               | リファクタリング判断者 / テックデット解消担当 / スプリント計画者 |
| `04-skills-in-practice/06-baoyu-diagram.md`         | アーキテクト / テックライター / 説明資料作成者                   |
| `04-skills-in-practice/07-baoyu-infographic.md`     | コンテンツ制作者 / 社内勉強会主催者 / マーケター                 |

### Tier 2（次点）: スキル作成・フレームワーク章

概念章だが「自分でスキルを作る」という行動に接続するシナリオが有効。

| ファイル                                          | シナリオの軸                                                    |
| ------------------------------------------------- | --------------------------------------------------------------- |
| `01-skill-creation/01-what-are-agent-skills.md` | 自動化したいワークフローを持つ開発者                            |
| `01-skill-creation/04-best-practices.md`        | スキルを作ったが品質に自信がない人                              |
| `03-frameworks/01-superpowers.md`               | エージェントが暴走した経験がある人 / チーム導入を検討しているTL |

### Tier 3（対象外）: 手順書・ツール紹介系

`02-discovery/`・`05-content-creation/`・`07-document-workflow/` 配下は手順の明確さの方が価値が高く、シナリオの追加効果が低いため対象外。

## 既存計画との関係

既存の品質向上計画（[2026-06-27-quality-improvement-design.md](./2026-06-27-quality-improvement-design.md)）は「Why/When セクション追加」を扱う。本仕様はそれとは独立した改修であり、干渉しない。

両方を同一ファイルに適用する場合の章構成（grill-me.md の例）:

```
## このスキルについて      ← 既存
## こんな状況に刺さる      ← 本仕様で追加（新規）
## Phase A: ...            ← 既存
...
## 設計の意図              ← Why/When 計画で追加済み
## この設計を変えるとき    ← Why/When 計画で追加済み
## 次のステップ            ← 既存
```
